概念 コンセプト5本柱 ─ セクション対応表

各記事は「商品・どんな人か・想い・評判・遷移」の5本柱が揃うことで、
顕在層の信用づくりと公式サイト遷移（CVR計測）を両立します。
validate-articles.js がこの表のフィールドを自動チェックします。

柱 目的 対応セクション 対応フィールド（主要） 検証
商品 サービス内容・強みの具体的な説明 03 記事本文 / 08 サービス詳細 featureBoxes, serviceCards, steps 警告
どんな人か 代表・人物の見える化。氏名検索で「誰がやっているか」が伝わる 02 記者紹介 / 05 インタビュー / 07 切り込みQ&A editorImg, editorName, interviews, cuttingQA 警告
想い 設立背景・ストーリー性・意外性の描写 10 創業ストーリー storyText 警告
評判 口コミ閲覧欲・第三者性。ネガ口コミも公平に扱う 05 インタビュー / 06 口コミ・レビュー / 04 スコア評価 reviews, interviews, scoreDetails 警告
遷移（CTA） 顕在層を公式サイトへ誘導。CVRとして計測する 11 CTA（全セクション内のボタン） officialUrl, ctaBtn 警告
取材ヒアリング雛形： _prompts/article-draft.md に5本柱に対応したヒアリングシートと AI プロンプトが格納されています。 取材前に印刷またはコピーして活用してください。
CVR計測について： 公式サイトへの遷移は UTM パラメータ付きで計測してください。設定方法は docs/outbound-tracking-guide.md を参照。
officialUrl が # のままだと計測できません（validate で警告が出ます）。
01 ヒーローセクション 必須
フィールド名説明推奨文字数必須
heroTag タグバッジ（例: "取材件数127件！" "SNSで話題沸騰！"） 〜15字 推奨
heroTitle h1見出し。<br> タグで改行可。サービス名 + 評判訴求を含めること 20〜35字 必須
heroSub サブコピー。取材規模・ネガ口コミへの言及を含めると効果的。記事一覧のエクサープトとして使用される 80〜120字 推奨
書き方のコツ：heroSub には「◯◯名に直接取材」「ネガコメントの真相も含め」などの表現を入れると読者の信頼感が増します。
02 記者紹介 / この記事で分かること 推奨
フィールド名説明推奨文字数必須
editorName記者名（例: "記者：漆沢 祐樹"）固定推奨
editorTitle記者肩書き〜30字推奨
editorImg記者顔写真URL（64×64px / 正方形）URL推奨
checkItems「この記事で分かること」リスト。3項目固定。読者の疑問に直接答える形で書く各20〜40字推奨
editorImg（記者顔写真）
64 × 64 px
正方形 / JPG or PNG
1枚
03 記事本文（記者コメント / 注目ポイント / 記者見解） 推奨
フィールド名説明推奨文字数必須
reporterNote 記者コメントカード。取材で最も印象に残った1点を主観的に書く。読者の続読を促す"フック"として機能する 60〜100字 推奨
featureBoxes 注目ポイント4点（01〜04番号付き）。取材で確認した根拠・事実を中心に書く。各タイトル15字以内、本文100〜150字 各100〜150字 推奨
journalistTake 記者の総括見解ボックス（金色ボーダー）。ポジティブ・ネガティブ両面と読者へのアクションアドバイスを含めること 80〜150字 推奨
注意： featureBoxes の4項目はすべて「取材で確認した事実」に基づいて書くこと。主観的な誇張表現は避け、根拠となる数字や具体的なエピソードを入れることを推奨します。

04 スコア評価バー（5指標） 推奨
フィールド名説明必須
scoreDetails[].label評価指標名（例: "サービス品質" "コスパ" "サポート体制" "透明性・誠実さ" "利用者満足度"）推奨
scoreDetails[].scoreスコア（数値 / 小数点1桁まで）推奨
scoreDetails[].max最大スコア（通常 5）推奨
scoreNoteスコア下の注釈（算出根拠・取材人数等）推奨
注意： 5指標の名称は原則固定。指標を変更する場合は社内承認が必要です。スコアは取材結果に基づく主観的評価であることを scoreNote に明記してください。
05 取材インタビュー（利用者3名） 推奨
フィールド名説明推奨文字数必須
interviews[].name取材協力者名（匿名可 / 例: "田中 M様（32歳・女性）"）〜20字推奨
interviews[].bg利用前の状況・職業・課題（1行）〜40字推奨
interviews[].resultType"success" または "other"（バッジの色が変わる）固定推奨
interviews[].resultLabelバッジに表示する結果ラベル（例: "転職成功 / 年収+80万"）〜20字推奨
interviews[].qaQ&Aペア（2問固定）。Q: 記者からの質問、A: 利用者の回答（一人称で、NG: 美化・誇張）A: 各60〜120字推奨
interviews[].obs取材後記（記者の補足コメント）。斜体で表示される50〜100字推奨
3名の構成推奨：
① 最もポジティブな成果を出した利用者（resultType: "success"）
② 別の属性・背景の成功事例（resultType: "success"）
③ 転職しなかった・別の結果など異なる角度のケース（resultType: "other"）
→ 多様な読者に自分ごと化させる構成が有効です。
06 口コミ・レビュー（5件） 必須
フィールド名説明必須
reviews[].id一意のID（連番: 1〜5）必須
reviews[].name投稿者名（例: "田中 M様"）必須
reviews[].age年代・性別（例: "32歳・女性"）必須
reviews[].stars評価点数（数値 1〜5。文字列 "5" は無効）必須
reviews[].tag特徴タグ（例: "転職成功" "効果実感"）必須
reviews[].textレビュー本文（実際の利用者の言葉で。記者による改変は内容・文脈を変えないこと）必須
重要： stars は必ず 数値（integer） で入力してください。文字列 "5" ではスコア計算が動きません。

フィールド名説明推奨文字数必須

cuttingQA[].who質問者（通常 "記者"）固定推奨
cuttingQA[].q記者からの鋭い質問（ネガコメント・疑問点に切り込む内容）30〜60字推奨

Q3問の構成推奨：
① ネガコメント・疑問点への直接質問（「〇〇という口コミがありますが…」）
② 具体的な条件・数字への確認（料金・解約・実績等）
③ 他社との差別化・競合比較への見解

09 FAQ（よくある疑問） 必須
フィールド名説明推奨文字数必須
faqs[].q質問文（読者が実際に疑問に思うことを想定して書く）20〜40字必須
faqs[].a回答文（具体的・誠実に。最後のFAQは「この記事は広告ですか？」を固定）60〜120字必須
固定ルール： FAQの最後（5問目）は必ず「この記事は広告ですか？」+ PR開示回答を入れること。景品表示法のステルスマーケティング規制対応のため必須です。

フロー 記事投稿フロー（ステップ別）
1

取材・コンテンツ素材の準備

利用者3名へのインタビュー（オンライン可） / 企業担当者へのQ&A確認 / 写真素材の取得・許諾確認。写真は images/ フォルダに格納しておく。

2

CLIウィザードで記事の骨格を生成

npm run add-article を実行。スラッグ・企業名・カテゴリ・公開日・ヒーロータイトル・サブコピー・公式URLを入力すると、data/articles.js に記事オブジェクトが自動追記される。

3

data/articles.js に詳細フィールドを追記

エディタで data/articles.js を開き、追記された記事オブジェクトに以下を追加する：
reporterNote / featureBoxes / journalistTake / scoreDetails / interviews / cuttingQA / cuttingSummary / reviews / galleries / serviceCards / steps / faqs / storyText

4

ローカルでプレビュー確認

python3 -m http.server 8080 または npx serve . でサーバーを起動し、http://localhost:8080/article.html?id=<slug> で記事を確認する。

5

バリデーション実行

npm run validate を実行し「✅ 検証完了: エラーなし」を確認する。エラーがある場合は修正してから次へ進む。

6

git commit & push → 自動デプロイ

git add data/articles.js images/ → git commit -m "記事追加: [企業名]の評判記事" → git push origin main
GitHub Actions が自動で検証 → GitHub Pages へデプロイ（約1〜2分）。

7

公開後の確認

gh run list --limit 3 でデプロイ状況を確認。本番URLで記事・トップページへの反映をチェックする。
